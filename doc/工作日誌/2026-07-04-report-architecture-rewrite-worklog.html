今天審查過程中抓到並修正的 9 個真實問題

1
Task 1：文件與實際不符 + 簡體字
清單文件寫「~100 檔」實際只有 65 檔；未使用的 import；「十铨」誤用簡體字（應為「十銓」）。
2
Task 2：ASX ADR 比例錯誤（1:5 → 1:2）
經 WebSearch 查證 SEC 20-F 申報文件與 Nasdaq 掛牌頁面，日月光投控 ADR 正確比例是 1:2（1:5 是 2003 年上市舊比例）。 不修正的話 ASX 的 ADR 溢價訊號會系統性錯誤約 2.5 倍。TSM、UMC 的 1:5 比例查證後仍正確。
3
Task 3：重複的 to_int() helper
fetch_margin_trading() 內重複定義了跟 _fetch_twse_t86() 一樣的 helper，已抽成共用的 _twse_to_int()。
4
Task 4：null 欄位讓整批資料靜默丟失
dict.get(key, "") 只在 key 不存在時生效，TWSE 回傳 null 值會讓 .strip() 拋 AttributeError，被外層 except 吃掉後整批資料消失。已改用 (row.get(key) or "")。
5
Task 6：NaN 過濾造成兩個 list 位置錯位
Close／Volume 各自過濾 NaN，若 NaN 出現在不同列，同一個 index 會對應到不同交易日。 已改成逐列一起過濾。這個檔案裡的 _has_nan_close 是真實踩過 yfinance NaN 問題後加的既有防線，佐證此風險並非理論性。
6
Task 7：除以零讓整個排程崩潰
score_rs_rank_signal() 對 closes[-21] 沒防零值，遇到停牌股會 ZeroDivisionError， 在還沒產出任何報告前就讓每日排程整個崩潰——比現有「HTML 驗證失敗就不發布」更早、更嚴重。已加零值防呆。
7
Task 7：供應鏈訊號後蓋前，丟失觸發理由
同一檔台股被兩個美股觸發時（例如 NVDA 跟 AMD 都對應緯創 3231），後蓋前會丟失第一個來源的說明文字， 影響 Task 9 AI 寫的「入選理由」正確性。已改成串接所有觸發來源。
8
Task 8：歷史檔案格式防呆不足
load_signal_history() 只防 JSON 語法錯誤，沒防「格式對但形狀不對」； compute_win_rate_review() 對手動可編輯檔案裡的 pick["code"] 直接存取，缺欄位會讓函式崩潰。 已加 isinstance(dict) 檢查與 .get() 防呆，並補 5 個先前沒覆蓋到的測試。
9
