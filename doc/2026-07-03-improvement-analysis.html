專案改善分析

Investment Report 2026 — 每日投資情報自動化 pipeline 的體檢報告（2026-07-03）
總覽
專案整體設計清楚：GitHub Actions 排程 → Claude API + web_search 生成 HTML → GitHub Pages 部署 → Telegram / LINE / Email 通知。 本文件記錄一次完整的程式碼審視結果，共 12 項改善建議，按影響程度分為「高影響（可靠性與正確性）」、「中影響（維護性）」、「低影響（整理）」三級。

現況做得好的地方

用 yfinance 預先抓取財報日曆、P/E、VIX 等結構化資料注入 prompt，避免 AI 憑空編造數字
Streaming 模式 + pause_turn 迭代迴圈，正確處理多輪 web_search 的長時間請求
以 Backup/{日期}.html 是否存在做防重複機制，一天只發布一次
美股假日判斷（exchange_calendars XNYS），假日自動跳過生成
通知步驟在 secrets 未設定時自動 no-op，且只在實際有 commit 時才發送

高影響 — 可靠性與正確性

1
每天讓 AI 從零重新生成整份 HTML（最大的結構性問題）
scripts/generate_report.py 每天要求模型輸出 100KB+ 的完整 HTML（含所有 CSS / JS / Chart.js 設定）。代價有三： 版面每天都可能長得不一樣、JS 有機會寫壞導致圖表掛掉、每天消耗接近 64K 輸出 token。
建議：把 HTML 模板與圖表程式碼固定放在 repo，讓 Claude 只輸出各區塊的結構化 JSON（數據 + 文字），再由 Python 注入模板。 版面永遠一致、輸出 token 大幅降低、資料格式也變得可驗證。這一項會同時讓第 2、5 項變得容易實作。
2
截斷的報告會直接發布上線
generate_report.py 第 416–430 行：遇到 max_tokens 截斷時，補上 </body></html> 就照樣寫入 index.html 並推上正式網站，使用者會看到半份報告。
建議：發布前做基本驗證 — 必要區塊是否存在、長度是否合理、HTML 是否可解析。不合格就讓 workflow 失敗、保留昨天的版本。
3
生成失敗時完全沒有通知
通知只在成功 commit 後發送。若生成失敗（API 錯誤、yfinance 掛掉、截斷），只能靠「沒收到訊息」才發現。
建議：在 daily-update.yml 加一個 if: failure() 的 Telegram / LINE 失敗告警步驟。成本極低、價值很高。
4
高風險精確數據仍依賴 web_search
法人連三日買賣超排行、台指期夜盤這類精確數字，讓 AI 上網搜尋是最容易出錯（甚至幻覺）的部分。
建議：證交所（openapi.twse.com.tw）與期交所都有免費 JSON API，比照財報日曆的做法預先抓取後注入 prompt，讓 AI 只負責解讀而不是找數字。
5
P/E 歷史趨勢的算法不是真的歷史 P/E
fetch_pe_history() 用「當前 trailing P/E 反推的固定 EPS」除歷史股價，等於假設 3 年來 EPS 不變 — 畫出來的只是股價走勢的縮放，不是真實估值變化。對 NVDA 這種 EPS 暴增的股票失真特別大。
建議：改用 yfinance 的季度 EPS 資料，計算各期真實 TTM EPS 後再算歷史 P/E。

中影響 — 維護性

6
報告重複存三份，且 archive/ 日期標錯一天
每天的報告同時存在 index.html、archive/、Backup/。 而且 generate_report.py 是把昨天的報告備份成 archive/{今天日期}.html，日期標錯一天；Backup/ 才是正確日期。 每天 ~100KB 進 git 歷史，一年會膨脹 25MB+。
建議：合併為一個歸檔目錄、統一命名規則（修正日期偏移），並考慮舊報告不進 git（或放 orphan branch）。
7
依賴沒有鎖版本
Workflow 每天安裝最新版 yfinance（出了名的常改 API、常壞）、exchange-calendars 等，某天 pipeline 可能無預警掛掉。
建議：新增 requirements.txt 鎖定版本，workflow 改用 pip install -r requirements.txt。
8
只檢查美股假日，沒檢查台股假日
is_prev_us_day_holiday() 只用 XNYS（紐約）日曆，但報告一半內容是台股。台股放假（農曆年、颱風假）時報告照跑，會拿到過期或錯誤的台股數據。
建議：exchange_calendars 也有 XTAI（台灣）日曆，可一併判斷，或在報告中標註「台股休市」。

低影響 — 整理

9
AGENTS.md 是壞掉的複製品
內容被全域取代成「Codex API (Codex-sonnet-4-6)」這種錯誤描述，會誤導其他 AI 工具。建議修正內容或直接指向 CLAUDE.md。
10
CLAUDE.md 的 Secrets 表格漏了 LINE
Workflow 已支援 LINE_CHANNEL_ACCESS_TOKEN / LINE_USER_ID / LINE_GROUP_ID（含多群組逗號分隔），但文件沒跟上。
11
git 工作區不乾淨
4 個舊報告被移到 Backup/ 但尚未 commit；CLAUDE.md、AGENTS.md、doc/ 皆為 untracked。建議整理後提交一次。
12
通知內容可以更有價值
目前 Telegram / LINE / Email 只發連結。可讓生成流程順便輸出 2–3 行當日重點（大盤漲跌、最重要事件）直接放進訊息，不點開也能掌握概況。

優先順序總表

項目等級主要效益工作量
#1 模板 + JSON 資料分離高穩定性、成本、可驗證性三者同時改善大
#2 發布前 HTML 驗證高避免半份報告上線小
#3 失敗告警通知高pipeline 掛掉時第一時間知道極小
#4 TWSE / 期交所 API 預抓高消除法人 / 期貨數據幻覺風險中
#5 真實 TTM EPS 計算 P/E高估值圖表不再失真中
#6 歸檔目錄整併 + 日期修正中repo 不再膨脹、歸檔日期正確小
#7 requirements.txt 鎖版本中避免依賴更新造成無預警故障極小
#8 台股假日判斷中台股休市日不產生錯誤數據小
#9–12 文件與整理低文件正確、工作區乾淨、通知更有用小
🎯
如果只做一件事
先做 #1 模板 + JSON 資料分離：它同時解決穩定性、成本、可驗證性三個問題， 而且做完之後 #2（驗證）和 #5（P/E 修正）都會變得容易許多。
