TROUBLESHOOTING

GitHub Actions 排程延遲問題與解法

說明為何自動排程不準時，以及如何透過外部 cron 服務確保準點發布

問題說明

⚠️
GitHub Actions 排程不保證準點觸發
GitHub Actions 的 schedule（cron）觸發器使用共用排程器。在高峰時段，排程器可能排隊積壓， 導致實際執行時間比設定時間延遲數分鐘至數小時，甚至在少數情況下完全跳過當次執行。 這是 GitHub 的平台限制，無法從 workflow 設定內部解決。

常見症狀：

!
排程延遲 1–3 小時

設定 08:00 執行，實際卻在 10:00 甚至 11:00 才觸發，Actions 頁面顯示「Scheduled」但時間偏移。

!
當天完全未執行

Actions 頁面沒有當天的排程紀錄，排程器在高峰時直接跳過該次執行。

!
repo 閒置 60 天後自動停用

GitHub 會停用超過 60 天沒有 push 活動的 repo 的排程 workflow，需手動至 Actions 頁面重新啟用。

根本解法：外部 cron 服務觸發

🎯
改用外部服務呼叫 GitHub API
捨棄 GitHub 自己的排程器，改由可靠的外部 cron 服務在指定時間呼叫 GitHub API， 觸發 workflow_dispatch 事件。外部服務負責計時，GitHub 收到請求後立即執行，準時率接近 100%。

推薦使用 cron-job.org（完全免費，無需信用卡），設定步驟如下：

1
建立 GitHub Personal Access Token（PAT）

前往 GitHub → Settings → Developer settings → Personal access tokens → Tokens (classic)， 點 Generate new token，勾選 workflow 權限，複製產生的 token。

2
註冊 cron-job.org 免費帳號

前往 cron-job.org 完成免費註冊，無需信用卡。

3
建立新的 Cronjob

點 Create cronjob，依下表填入設定：

URL https://api.github.com/repos/jason-hey/investment-report-2026/actions/workflows/daily-update.yml/dispatches
Request method POST
Header: Authorization Bearer <你的 GitHub PAT>
Header: Accept application/vnd.github+json
Header: Content-Type application/json
Request body {"ref":"main"}
Schedule（時區選 Asia/Taipei） 每週二、三、四、五、六 — 08:00
4
移除或保留 GitHub 原本的 schedule

可以從 daily-update.yml 中刪除 schedule 區塊，只保留 workflow_dispatch， 避免兩者重複觸發。或保留作為備援（重複觸發也只會多一次執行）。

5
驗證設定

在 cron-job.org 手動執行一次，確認 GitHub Actions 頁面出現新的 workflow_dispatch 執行紀錄， 且狀態為成功。

注意： GitHub PAT 是敏感資訊，請勿將其寫入 repo 或公開分享。 cron-job.org 的 header 欄位是加密儲存的，安全性足夠。

目前 Workflow 排程設定

.github/workflows/daily-update.yml 目前的 cron 設定： 

on:
  schedule:
    # 台灣時間 08:00 = UTC 00:00（週二至週五、週六）
    - cron: '0 0 * * 2-6'
  workflow_dispatch:  # 支援手動觸發
    inputs:
      date_override:
        description: '指定日期 YYYY-MM-DD（留空=今天）'
        required: false
cron 語法說明： 0 0 * * 2-6 — 每週二（2）至週六（6）的 UTC 00:00（= 台灣時間 08:00）執行。 數字 0=週日、1=週一、2=週二、…、6=週六。
